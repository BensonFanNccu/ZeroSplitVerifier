# ZeroSplitVerifier 可再現性審查報告(完整版)

> 受審對象:林立仁(leolin-hub)2026 年論文《Robustness Verification of RNN with Abstraction Refinement》及其程式碼。
> 審查者:獨立第三方。目的:判斷這份程式能否、並在何種程度上重現論文第五章的實驗結果。
> 本檔為**單一總報告**,彙整全部實驗數據與發現。審查全程未更動受審程式碼(`master` 與上游一致),產物皆置於倉庫之外。
> 完成於 2026-06-30。GenBaB(論文表 5.5、圖 5.4d)依議定不納入。

---

## 1. 摘要與總體結論

- **方法可重現、結論成立。** 三張純循環網路表(5.1/5.2/5.3)在自行重訓下即與論文趨勢相符(救回率平均誤差 5–11 個百分點),
  改用作者釋出的模型後更貼近論文(誤差在 CIFAR-10 與筆畫序列上約減半);論文主張「救回率隨序列長度下降」清楚重現。
- **先前無法逐格 exact 的主因是訓練權重不可重現;作者的模型釋出(`v1.0-artifacts`)大幅彌補了這點。** 以作者模型重跑,
  救回數幾乎吻合論文,證實落差來自每次重訓得到不同權重,而非方法或實作有誤。
- **照說明文件直接執行仍會失敗:** 內附批次腳本預設網格與論文不符、且多進程設計在 Linux 會死結。這些屬程式/文件層面,
  作者目前以「附權重 + 文件」繞過而非修碼。
- **長短期記憶網路表(5.4)無法 exact 重現:** 訓練不可重現的影響在此被放大,且 `v1.0-artifacts` 未含標準長短期記憶模型;
  其定性結論(細化有效、長短期記憶遠慢於純循環)成立。

---

## 2. 審查背景與目標

ZeroSplit 是建立於 POPQORN 之上的抽象細化驗證框架:當線性鬆弛的抽象基準因近似誤差無法認證某個本為強健的樣本時,
挑出誤差最大的神經元,將其跨零的預啟動區間切成正、負兩段、各自以更緊的邊界重算,兩段皆認證才回報「已認證」。
本審查不評論方法的學術貢獻,只回答:**換一個人、在合理努力下,這份程式能否重現論文第五章的數字?**

---

## 3. 審查方法與原則

- **唯讀:** 受審倉庫 `master` 維持與上游一致(可 `git diff` 驗證);所有重現產物在另一分支或倉庫之外。
- **重現策略:** 不使用內附 `auto_test_*.py`(預設網格與論文不符),改自寫外部驅動逐格呼叫核心驗證器。
- **參數對齊論文第五章:** L2 範數、每格 50 樣本、擾動 0.005→0.1(步長 0.001)、細化深度 5(對應「至多 31 次細化」)、CPU。
- **指標:** 論文的「觸發數 ×」=基準在某擾動半徑首次失敗的樣本數;「救回數 ↑」=其中經細化救回者;「救回率 %」=↑/×。
- **作者 artifacts:** 審查期間作者發布 `v1.0-artifacts`(`models.zip` 112 模型 + `test_samples.zip`,附 SHA-256)。
  我們**通過 SHA-256 校驗**後,額外以「作者模型」重跑三張 RNN 表做準-exact 比對(見第 5–6 節)。

論文對環境幾無記載。經確認,論文實驗與本審查在**同一台實體電腦、不同帳號**進行,故硬體(32 核、約 91GB)相同;

可能不同,只比趨勢。**

配置耗時 22 分鐘至 9.5 小時,印證「長短期記憶比純循環慢 2–3 個數量級」。

**補充:以作者的 ReLU-LSTM 重跑(回應作者「表 5.4 係以 ReLU-LSTM 進行」之說法)。** 作者的 `v1.0-artifacts` 未含
標準長短期記憶,但提供了 ReLU-LSTM(g 閘用 relu、i/f/o 閘用 sigmoid、輸出 identity)。我們以 `--relu` 模式載入
作者的 ReLU-LSTM 重跑全部 16 配置(15 完成、1 崩潰,見下),結果**方向上支持作者說法,但仍非乾淨重現**:

- ReLU-LSTM 較標準長短期記憶**更接近論文**(平均 \|Δ×\| 由 24.2 降到 **16.5**),且**長序列 m=7 的觸發數幾乎完全吻合
  論文的 50**(50/50、50/50、49/50)。
- 但**小序列 m=1、2 的觸發數仍大幅低於論文**(如 m=2、隱藏層 32:我們 ×2 對論文 ×34),平均救回率誤差仍達 20.4
  個百分點——與三張 RNN 表以作者模型達到 \|Δ×\|≈0 的準-exact 截然不同。
- **結論:即便使用作者的確切 ReLU-LSTM 權重,表 5.4 仍是最難重現的一張。** 殘差成因未能完全釐清(可能為 release
  的 ReLU-LSTM 非表 5.4 當時之確切權重,或長短期記憶的 2D 閘-細胞邊界運算對軟體/數值環境較敏感)。

- **穩健性問題(新發現):** 配置 h16/m4 在驗證中丟出例外
  `x_plus must be strictly larger than x_minus and y_plus must be strictly larger than y_minus`
  (來自 2D 閘-細胞邊界 `BoundTanhSigmoidy` 遇到退化的零寬區間),導致該配置整個崩潰。這是驗證器在特定配置下的
  穩健性缺陷,建議對零寬/退化區間加防護。

## 6. 作者 `v1.0-artifacts` 的影響(準-exact 重現)

審查期間作者發布的 `v1.0-artifacts` 提供 112 個訓練好的模型(三張 RNN 表各 32 + ReLU-LSTM 16)與固定樣本。以作者模型
重跑三張 RNN 表(96 配置全部完成),數值一致地較自行重訓更接近論文(見 5.1 總表),**單格最明顯者**為序列長度 7、
隱藏層 8、ReLU:作者模型 ×29/↑17/59%、論文 25/16/64%、自訓僅 ↑9。**這在控制變因下證實了第七項發現(訓練不可重現)
是先前數值落差的主因。** 限制:此 release **未含標準 sigmoid/tanh 長短期記憶模型**(只給 GenBaB 用的 ReLU-LSTM),
故表 5.4 無法以作者模型改善。

## 7. 主要發現:論文與程式之間的落差

1. **取樣方式不一致。** 論文§5.1.2 稱「前 50 個」,程式實為 `seed=2025`+`shuffle=True` 的隨機 50 個
   (`vanilla_rnn/utils/sample_data.py:10-16`)。固定環境下可重現,但敘述與實作不符。
2. **擾動掃描範圍(僅批次腳本)。** 各 `auto_test` 預設範圍互異;惟步長 0.001 內建於驗證器、與論文一致,範圍可由命令列調整,已解。
3. **批次腳本網格與論文不符。** `auto_test_*.py` 預設為縮減版開發配置,故須自寫外部驅動。作者已在 README 修正網格數值,但碼內預設未改。
4. **細化深度。** `--max-splits` 為遞迴深度,論文「31 次細化」=深度 5;統一採深度 5,已釐清。
5. **訓練不可重現(致命,已由附權重緩解)。** 所有 `train_*.py` 無隨機種子,論文未報超參數/準確率 → 每次重訓權重不同。
   作者釋出模型後可不必重訓即重現;若再於訓練腳本設種子則更徹底。
6. **軟體環境與種子未記載。** 硬體雖與審查同機,但作者軟體堆疊與種子已不可考 → 執行時間絕對值不保證可比。
7. **序列長度數字三方不一致。** 筆畫序列:訓練腳本預設 55、README 寫 50、論文用 30–45。
8. **多進程死結(使用性影響大)。** 兩驗證器 `mp.Pool`(`rnn_zerosplit_verifier.py:669`、`lstm_zerosplit_verifier.py:417`)
   未設啟動方式 → Linux 預設 fork;在 PyTorch/OpenMP(LSTM 再加 numba)多執行緒後 fork 會死結。**照預設用多工作程序會卡死。**
   暫解:把 `OMP_NUM_THREADS`/`MKL_NUM_THREADS`/`OPENBLAS_NUM_THREADS`(LSTM 再加 `NUMBA_NUM_THREADS`)設為 1;根本解為 `spawn`。

9. **驗證器穩健性缺陷(新)。** 以作者 ReLU-LSTM 跑表 5.4 時,配置 h16/m4 丟出例外
   `x_plus must be strictly larger than x_minus...`(2D 閘-細胞邊界 `BoundTanhSigmoidy` 遇零寬退化區間),
   整個配置崩潰。建議對退化區間加防護(clamp 或跳過)。

**有利於可再現性的客觀事實:** 核心驗證器命令列參數完整(使「不改碼重現論文網格」可行);LSTM 最佳分裂點查表
**與模型無關**(我們從零重建 tanh/sigmoid 兩表,與倉庫所附位元級完全相同);提供 `--toy-rnn` 零依賴自測與
`package_artifacts.py` 可重現打包。
